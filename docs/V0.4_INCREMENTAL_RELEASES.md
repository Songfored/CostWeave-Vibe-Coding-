# CostWeave v0.4 增量发布列车

> 文档状态：实施基线  
> 总体蓝图：`V0.4_UPGRADE_ROADMAP.md` R3  
> 当前基线：v0.4.1（已完成）  
> 目标：通过 v0.4.1～v0.4.9 九个稳定版本完成 v0.4 总体能力，而不是一次性重写系统

## 1. 结论

R3 是完整目标架构，不应作为单次版本的开发清单。v0.4 采用增量发布列车：

```text
v0.3.0
  ↓
v0.4.1 合同与状态守卫
  ↓
v0.4.2 持久化控制内核
  ↓
v0.4.3 Artifact 与任务图
  ↓
v0.4.4 并行运行时与准入
  ↓
v0.4.5 成本经济与资源组合
  ↓
v0.4.6 证据验收与分级恢复
  ↓
v0.4.7 结果预测与评估闭环
  ↓
v0.4.8 图像/视频制作工作流
  ↓
v0.4.9 整合、加固与 v0.4 完成版
```

每个版本必须：

- 可以单独启动和演示
- 有明确用户价值
- 只引入一个主要架构风险
- 保持模型目录和既有数据可用
- 有升级、重启和回滚测试
- 达到自己的退出门槛后才进入下一版本

本项目处于 `0.x` 阶段，使用 `v0.4.1`～`v0.4.9` 表示同一 v0.4 发布列车中的功能里程碑；`v0.4.9` 是 R3 蓝图在离线/仿真边界内的完成版。

## 2. 拆分原则

### 2.1 先控制正确，再增加智能

开发顺序必须是：

```text
合同和状态边界明确
→ 状态可持久恢复
→ 交接可靠
→ 并行可靠
→ 成本决策
→ 失败恢复
→ 结果预测
→ 垂直媒体工作流
→ 产品加固
```

如果事件、状态和 Artifact 还不可靠，后面的概率预测和媒体编排只会放大错误。

### 2.2 每个版本只有一个状态所有者

从 v0.4.1 开始冻结状态变更合同；从 v0.4.2 开始由持久化控制内核成为唯一状态所有者：

- v0.4.1 的兼容外壳统一校验状态转换，但不改写 v0.3 运行时
- v0.4.2 起只有 `ControlKernel` 能提交状态变更
- LLM、Planner、Router、Validator 和 UI 只能提交命令或建议
- v0.4.2 起 `EventStore` 是运行状态的唯一事实来源
- Artifact 修改产生新版本，不覆盖旧内容
- 模型目录继续使用运行快照隔离

### 2.3 兼容优先

- v0.3 API 在 v0.4.1～v0.4.8 保持兼容层
- 新能力使用 `/api/v1`
- 模型目录 Schema 不因运行时升级被破坏
- 每个数据库迁移都有版本号、前置检查和失败回滚
- 新功能默认通过 Feature Flag 启用

建议功能开关：

```text
COSTWEAVE_CONTRACTS_V4
COSTWEAVE_RUNTIME_V4
COSTWEAVE_ARTIFACT_DAG
COSTWEAVE_ADMISSION_CONTROL
COSTWEAVE_RESOURCE_OPTIMIZER
COSTWEAVE_OUTCOME_FORECAST
COSTWEAVE_MEDIA_WORKFLOW
```

### 2.4 稳定版本定义

一个版本只有同时满足以下条件才算完成：

1. 全新环境可以安装和启动。
2. 上一个版本可以无人工改库升级；没有 Schema 变化时明确标记“无需迁移”。
3. 本版引入的持久状态在服务重启后不会丢失；未引入持久状态时必须明确边界。
4. 本版新增的命令或请求不会重复执行或计费。
5. 旧功能回归测试全部通过。
6. 新功能有端到端测试。
7. 页面能明确展示仿真与真实数据边界。
8. 发布包、变更日志、迁移指南和验收报告齐全。

## 4. v0.4.1：Contracts & State Guard

### 4.1 目标

用最小改动冻结 v0.4 的核心数据合同和合法状态转换边界，为后续控制内核提供稳定接口，同时保持 v0.3 的运行逻辑、页面和模型目录不变。

### 4.2 用户可见价值

- 当前 v0.3 的全部功能和页面继续可用
- 运行状态不能再被任意写成非法值
- 调试信息能够显示合同版本、状态转换来源和拒绝原因
- GitHub 发布物能够清楚展示 v0.4 已从“自由对象传递”迈向版本化协议
- 不增加任何模型调用、数据库迁移或后台服务成本

### 4.3 实现范围

新增：

```text
contracts_v4.py
state_guard.py
compat_v4.py
```

实现：

- `RunId`、`TaskId`、`EventId` 和 `ArtifactId` 的统一生成与格式校验
- 带 `schema_version` 的 `CommandEnvelope` 和 `EventEnvelope`
- 显式运行/任务状态枚举和合法转换表
- 纯函数 `validate_transition(current, target)`，拒绝非法转换和终态回退
- v0.3 `RunRecord` 与 v4 合同之间的序列化兼容外壳
- 进程内只读事件踪迹，用于测试和调试，不充当持久化事实源
- `COSTWEAVE_CONTRACTS_V4` 功能开关；关闭时完全沿用 v0.3 行为

### 4.4 数据与 API

- 不新增数据库和迁移
- 不新增必需 API
- 现有健康信息可附加只读的 `contract_schema: "v4.1"` 与功能开关状态
- 新合同必须可以序列化为 JSON，并能无损反序列化
- 事件踪迹随进程结束而丢失，这是本版本明确边界，不得宣传为持久化

### 4.5 验收门槛

- 合法和非法状态转换测试全部通过
- 终态回退、未知状态和错误 Schema 拒绝率 100%
- 四类 ID 和两个 Envelope 的 JSON 往返结果一致
- 开关打开和关闭时，v0.3 的规划、模拟运行和模型目录行为均不回归
- 不新增数据库文件、不改变既有模型目录 Schema
- 现有 41 项测试继续通过，总测试数至少达到 50 项
- 在全新环境完成安装、启动和基础页面冒烟测试
- 发布说明明确标注“合同与守卫版，不具备重启恢复能力”

### 4.6 明确不做

- 不引入 SQLite、EventStore、迁移器或启动恢复
- 不替换 `RunManager` 的状态所有权
- 不新增调度器、并行执行、Artifact DAG 或成本算法
- 不引入新路由算法
- 不重新设计页面主工作流
- 不接真实模型

### 4.7 回滚

关闭 `COSTWEAVE_CONTRACTS_V4` 即回到 v0.3 行为。由于没有数据库迁移、外部副作用和持久化新格式，不需要数据回滚。

### 4.8 低额度实施边界

本版本按“本周额度优先”约束实施：

- 目标改动控制在 3 个小模块和少量兼容接线
- 只做确定性代码与测试，不让模型参与运行时判断
- 不为本版本制作复杂新 UI
- 若某个需求迫使修改调度器、存储层或大量现有接口，立即移入 v0.4.2
- 通过标准是“小而真实、可回归、可发布”，不是提前实现 v0.4 的智能能力

### 9.3 实现范围

## 16. v0.4 最终完成定义

完成 v0.4.9 后，CostWeave 应当能够在不接真实模型的默认环境中，可靠展示：

1. 接收用户目标并生成可验证任务图。
2. 冻结模型目录和策略修订。
3. 在执行前完成能力、成本、期限和资源判断。
4. 并行执行仿真任务，不重复计费。
5. 根据独立证据验收每个节点。
6. 对失败进行最小充分恢复。
7. 实时预测最终质量、成本和时间区间。
8. 保存可归因 Outcome，支持基准与影子比较。
9. 对图像/视频目标生成视觉圣经、镜头合同和媒体制作 DAG。
10. 完成一个带审批、并行、局部重渲染和成本账本的 30 秒仿真短片。

这时的 v0.4 是一个结构清晰、可验证、可以继续接入真实模型的 Adaptive Runtime，而不是把所有未来设想一次性塞入同一个不稳定版本。
